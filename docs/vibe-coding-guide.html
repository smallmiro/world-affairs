이슈 기반 개발 — 계획을 세우고 이슈로 관리한다

이 프로젝트의 핵심 워크플로우는 "GitHub 이슈"를 중심으로 돌아갑니다. 모든 작업은 이슈를 먼저 만들고, 이슈 단위로 개발하고, 이슈 단위로 리뷰/머지합니다.

1

마일스톤 계획

큰 목표를 마일스톤으로 나누고 순서를 정한다

2

이슈 생성

Tidy First 기준으로 구조/기능 이슈를 분리 생성

3

에이전트 개발

전문 에이전트가 이슈별로 브랜치 생성 후 개발

4

PR + 코드 리뷰

code-reviewer 에이전트가 자동 리뷰 후 머지

이슈에 담기는 정보

이슈는 단순한 "할 일 목록"이 아닙니다. 각 이슈에는 다음 정보가 포함됩니다:

PRD 참조
docs/prd.md 섹션 번호
담당 에이전트
feature-dev, code-architect 등
상세 설계
영향 파일, API, DB 스키마
테스트 계획
Red → Green → Refactor

PRD란 무엇인가 — AI에게 주는 프로젝트 설계도

PRD(Product Requirements Document)는 "무엇을 만들 것인가"를 정의하는 문서입니다. 비개발자가 바이브 코딩을 할 때 가장 중요한 산출물이며, AI가 이 문서를 기준으로 모든 작업을 수행합니다.

PRD의 구성 — 이 프로젝트의 실제 구조

PRD는 다음과 같은 섹션으로 구성됩니다. 처음부터 완벽하게 쓸 필요 없이, 대화하면서 점진적으로 채워갑니다.

1. 프로젝트 개요
목적, 배경, 타겟 사용자
2. 기술 스택
사용할 프레임워크, DB, API 목록
3. 핵심 기능 정의
각 기능별 상세 요구사항 (뉴스, 시장, 선박, 공항...)
4. UI/UX 요구사항
레이아웃, 컴포넌트, 반응형 설계
5. 마일스톤 계획
M1~M7 단계별 목표와 우선순위
부록: 데이터 소스
각 API의 URL, 인증 방식, 제한 사항

PRD → 이슈 → AI 개발 루프

PRD를 한 번 쓰고 끝이 아닙니다. PRD를 기반으로 이슈를 생성하고, 이슈 단위로 AI에게 일을 시키는 순환 구조가 핵심입니다.

PRD 작성→ 마일스톤 분리→ GitHub 이슈 생성→ 에이전트 개발→ 리뷰 + 머지→ PRD 갱신⟳
Step 1: PRD → 이슈 리스트 자동 생성

사용자가 "M1 데이터 수집 이슈를 만들어주세요"라고 하면, code-architect 에이전트가 PRD를 분석하여 이슈 #3~#10을 자동 생성합니다. 각 이슈에는 PRD 섹션 참조, 담당 에이전트, 상세 설계가 포함됩니다.

Step 2: 이슈 → 에이전트에게 작업 지시

이슈가 생성되면 "이슈 #4를 구현하세요"라고 지시합니다. feature-dev 에이전트가 이슈 내용을 읽고 브랜치 생성 → TDD 개발 → PR 생성까지 자율적으로 수행합니다.

Step 3: 리뷰 + 새 이슈 발견

code-reviewer가 PR을 리뷰하면서 개선점이나 누락된 요구사항을 발견합니다. 이것이 새 이슈로 등록되고, 다시 개발 사이클이 반복됩니다.

Step 4: PRD 갱신 → 루프 반복

개발 중 새 요구사항(예: 두바이 공항)이 추가되면 PRD를 갱신합니다. 갱신된 PRD를 AI에게 참조시키면, AI가 변경사항을 인지하고 새 이슈를 제안합니다.

실제 사례: 두바이 공항 요구사항 추가 → PRD 섹션 4.8 신규 작성 → code-architect가 이슈 #27~#34 자동 생성 → 3개 에이전트 Worktree 병렬 개발 → 약 2시간 만에 완성
핵심 인사이트

비개발자의 7가지 역할

바이브 코딩에서 사람은 코드를 쓰지 않지만, 7가지 역할을 수행합니다.

💡

역할 1: 기획자

"뭘 만들 건지 말하기" — 한 문장에서 시작해 대화하며 기획을 키운다
"국제 정세 대시보드를 만들고 싶어요" → "중동 유조선도" → "주식도" → "두바이 공항도"
📐

역할 2: 설계자

"AI의 작업 규칙 정하기" — CLAUDE.md로 일관된 품질 유지
TDD, Git-Flow, Tidy First, 전문 에이전트 목록까지 정의 → AI가 매번 같은 수준으로 작업
📋

역할 3: PM

"이슈를 만들고 순서 정하기" — 마크다운 계획서로 Phase/이슈 관리
"Phase 1(순차) → Phase 2(병렬 3개 worktree) → Phase 3(순차 통합)"
🎨

역할 4: 디자인 디렉터

"눈으로 보고 피드백" — 픽셀 단위까지 조정 가능
"뉴스 높이 640px" → "650px" → "670px" → "690px" / "Tokyo Night 풍으로"
🔑

역할 5: 데이터 관리자

"외부 서비스 연결" — API 키 발급, 가입은 사람의 몫
OpenSky, AISStream, Gemini, ACLED 등 10개+ 키 직접 발급 및 .env 설정
🔍

역할 6: QA 테스터

"안 되는 걸 찾기" — AI의 '완료' 보고를 검증
"EK 비행기가 없는데요" / "INSERT가 아니라 UPDATE여야" / "선박이 두 개뿐"
🖥️

역할 7: 운영자

"시스템을 띄우고 관리" — 명령어 몇 개로 서버 관리
"pm2 start ecosystem.config.js" / "pm2 재시작" / "캐시 다 날리고"
실제 대화 기록 분석
